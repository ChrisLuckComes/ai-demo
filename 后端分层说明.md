# resume-agent 后端分层说明

这份文档的目标不是讲框架语法，而是回答一个更重要的问题：

**现在这个后端为什么这样分层，每个文件该放什么，不该放什么。**

---

## 一、当前后端模块划分

目前后端核心文件如下：

- `server.py`
- `agent.py`
- `interview_agent.py`
- `agent_prompts.py`
- `agent_utils.py`
- `models.py`
- `database.py`
- `resume_parser.py`

这套分法的核心思路是：

- `server.py` 负责接口层
- `agent.py` 负责 AI 业务编排层
- `interview_agent.py` 负责模拟面试智能体编排层
- `agent_prompts.py` 负责 prompt 定义
- `agent_utils.py` 负责纯工具函数和降级逻辑
- `models.py` 负责数据模型
- `database.py` 负责数据库与 Redis 基础设施
- `resume_parser.py` 负责文档解析

这已经是一个比较健康的边界，适合当前项目规模。

---

## 二、每一层的职责

## 1. `server.py`：接口层 / API 层

文件：`D:\code\resume-agent\server.py`

### 这一层负责什么

- 接收 HTTP 请求
- 做基础参数校验
- 调用 `ResumeAgent` 完成业务
- 返回 HTTP / SSE 响应
- 管理应用启动与关闭资源

### 这一层不应该负责什么

- 不应该写复杂 prompt
- 不应该写检索逻辑
- 不应该写模型输出清洗逻辑
- 不应该写向量切片策略

### 你可以把它理解成

它是“门面层”或“路由层”。

外部请求先进这里，真正的 AI 逻辑再交给 `agent.py`。

---

## 2. `agent.py`：AI 业务编排层

文件：`D:\code\resume-agent\agent.py`

### 这一层负责什么

- 初始化 LLM、embedding、vector store、OCR、parser
- 编排简历入库流程
- 编排 JD 分析流程
- 编排简历评估流程
- 编排聊天流程
- 管理缓存和历史消息读取
- 控制“检索 -> prompt -> 模型 -> 清洗 -> 返回”的主链路

### 这一层不应该负责什么

- 不应该堆大量 prompt 字符串
- 不应该塞太多无上下文工具函数
- 不应该写数据库底层连接逻辑

### 为什么 `agent.py` 仍然可以保留这些方法

因为这些方法都强依赖 `ResumeAgent` 的内部状态：

- `self.llm`
- `self.vector_store`
- `self.redis_client`
- `self.parser`

也就是说，它们不是纯函数，而是“围绕 agent 状态工作的流程函数”。

所以现在让 `agent.py` 做 orchestrator 是合理的。

---

## 3. `interview_agent.py`：模拟面试智能体编排层

文件：`D:\code\resume-agent\interview_agent.py`

### 这一层负责什么

- 复用 `ResumeAgent` 已经准备好的评估上下文
- 负责模拟面试问题生成与答案评分
- 负责面试问题缓存预热
- 负责把面试链路和主评估链路隔离，避免 `agent.py` 继续膨胀

### 为什么单独抽这一层

因为“模拟面试”已经不只是一个 prompt，而是一条独立子流程：

- 要依赖评估结果
- 要管理题目缓存
- 要分别处理出题和评分
- 还要维护 fallback

这类能力继续堆在 `agent.py` 里会让主 agent 失去边界，因此抽成单独的面试智能体更合理。

### 这一层不应该负责什么

- 不应该直接处理 HTTP 请求
- 不应该初始化底层模型和向量库
- 不应该承载通用工具函数

它的角色是：

**站在 `ResumeAgent` 之上，负责编排“模拟面试”这条独立 AI 子流程。**

---

## 4. `agent_prompts.py`：Prompt 定义层

文件：`D:\code\resume-agent\agent_prompts.py`

### 这一层负责什么

- 统一维护 system instruction
- 统一维护各种场景的 prompt 模板
- 让 prompt 变成显式、可维护、可替换的模块

### 当前包含的内容

- `SYSTEM_INSTRUCTION`
- `build_follow_up_prompt`
- `build_jd_keyword_prompt`
- `build_evaluation_prompt`
- `build_chat_prompt`

### 这样拆的好处

- 改 prompt 不会污染主流程代码
- 更容易做 prompt 版本化
- 更容易未来接 Prompt Playground
- 面试时能清楚讲“prompt 与业务逻辑解耦”

### 这一层不应该负责什么

- 不应该访问数据库
- 不应该直接调用模型
- 不应该做数据清洗

它只负责“定义输入模板”。

---

## 5. `agent_utils.py`：工具函数层

文件：`D:\code\resume-agent\agent_utils.py`

### 这一层负责什么

- 放可以被复用的纯工具函数
- 放和 `ResumeAgent` 状态弱相关的逻辑
- 放规范化、降级、格式转换函数

### 当前包含的内容

- 模型结果转换：`coerce_model`
- 消息转换：`to_langchain_message`
- 通用去重：`unique_strings`
- JD 降级提词：`fallback_keywords`
- 关键词标准化：`normalize_keyword_candidates`
- 雷达图构建：`build_radar_metrics`
- chunk 文本提取：`chunk_to_text`
- 聊天降级文案：`build_chat_fallback`
- 证据来源构建与格式化：
  - `build_resume_sources`
  - `format_sources_for_prompt`
  - `format_context_docs`
- 来源与评估项清洗：
  - `normalize_sources`
  - `normalize_source_ids`
  - `normalize_evaluation_items`
- 评估降级：`fallback_evaluation`

### 为什么这些适合放这里

因为它们大多满足这些特点：

- 输入明确
- 输出明确
- 副作用少
- 不依赖 `self.vector_store` 这类复杂状态

这意味着它们更适合被抽成独立函数。

---

## 6. `models.py`：数据模型层

文件：`D:\code\resume-agent\models.py`

### 这一层负责什么

- 定义数据库 ORM 模型
- 定义接口请求/响应模型
- 定义结构化输出模型

### 当前承载了三类模型

#### 1) 数据库模型

- `Resume`
- `ChatMessage`
- `InterviewSession`

#### 2) API 请求响应模型

- `QueryRequest`
- `EvaluationRequest`
- `JDAnalysisRequest`
- `JDAnalysisResponse`
- `ChatRequest`
- `OCRResponse`
- `ChatSuggestionsResponse`

#### 3) AI 结构化输出模型

- `RadarMetric`
- `EvidenceSource`
- `EvaluationItem`
- `EvaluationResult`
- `JDKeywordExtractionResult`

### 为什么这样合理

因为这个项目目前规模不大，把“所有 schema 放一起”比拆很多小文件更易维护。

如果后面项目再变大，再按：

- `db_models.py`
- `api_schemas.py`
- `ai_schemas.py`

继续细拆就行。

---

## 7. `database.py`：基础设施层

文件：`D:\code\resume-agent\database.py`

### 这一层负责什么

- 构造数据库连接 URL
- 初始化 SQLAlchemy engine / session
- 提供 `get_db`
- 初始化 Redis 客户端

### 这一层不应该负责什么

- 不应该写业务查询逻辑
- 不应该写 prompt
- 不应该写 AI 推理逻辑

它的角色就是：

**提供基础设施，不承载业务。**

---

## 8. `resume_parser.py`：文档处理层

文件：`D:\code\resume-agent\resume_parser.py`

### 这一层负责什么

- 文档解析
- 文本清洗
- chunk 切分

### 为什么不放进 `agent.py`

因为它本身就是一个独立能力：

- 输入文件
- 输出纯文本和 chunk

这类“文档处理能力”天然应该独立。

## 三、当前调用关系

你可以把整个后端调用链理解成这样：

```text
    HTTP Request
  -> server.py
  -> agent.py
     -> interview_agent.py
      -> agent_prompts.py
      -> agent_utils.py
      -> resume_parser.py
     -> models.py
     -> database.py / Redis / Chroma / LLM
  -> HTTP Response / SSE
```

如果按更具体的两条主链来看：

### 1. 上传简历链路

```text
server.py /upload_resume
  -> agent.ingest_resume()
  -> resume_parser.extract_text()
  -> resume_parser.get_chunks()
  -> Chroma.add_texts()
  -> 返回 resume_id / status
```

### 2. 评估简历链路

```text
server.py /evaluate
  -> agent.evaluate_resume()
  -> agent._extract_jd_keywords()
  -> agent_prompts.build_evaluation_prompt()
  -> LLM structured output
  -> agent_utils.normalize_*()
  -> 返回 evaluation
```

### 3. 聊天链路

```text
server.py /chat
  -> agent.stream_chat()
  -> _load_history()
  -> _retrieve_context_docs()
  -> agent_prompts.build_chat_prompt()
  -> LLM stream
  -> server.py 组装 SSE 事件
```

### 4. 模拟面试链路

```text
server.py /interview/start_stream
  -> interview_agent.prepare()
  -> agent.prepare_evaluation_context()
  -> interview_agent.generate_interview_questions()
  -> server.py 逐题返回 SSE

server.py /interview/submit
  -> interview_agent.submit()
  -> interview_agent.submit_interview_answers()
  -> 后端规则计算 verdict
```

## 四、什么叫“不过度拆分”

你刚才的判断是对的：

**不是拆得越细越高级。**

很多项目在规模还不大时，过早拆成：

- `chat_service.py`
- `evaluation_service.py`
- `prompt_service.py`
- `source_service.py`
- `cache_service.py`

表面上很“架构化”，实际上会带来问题：

- 跳文件太多
- 修改一条链路要来回切换多个模块
- 初学者更难读懂
- 面试时也不一定加分

### 当前这个项目更合理的平衡点是

- 主流程保留在 `agent.py`
- 独立子流程收敛到 `interview_agent.py`
- prompt 抽出去
- 工具函数抽出去
- 其余不要继续硬拆

这就是“不过度设计”的意思。

## 五、以后什么时候再继续拆

当出现下面这些信号时，再考虑下一轮拆分：

### 1. `agent.py` 再次膨胀到难以维护

比如：

- 700~1000 行以上
- 聊天、评估、入库逻辑彼此耦合严重

### 2. 某个子流程明显独立

例如：

- 评估逻辑需要支持很多模式
- 聊天逻辑要接入工具调用
- 文档解析要支持多格式、多策略

### 3. 需要单独测试某块复杂逻辑

例如：

- 想给“评估链路”写大量单测
- 想给“检索策略”单独做实验

这时再拆成：

- `evaluation_service.py`
- `chat_service.py`
- `retrieval_service.py`

会更自然。

## 六、你现在应该如何维护这个项目

我建议你后续新增代码时遵守这套规则：

### 放到 `server.py` 的内容

- 新接口
- 参数校验
- 响应拼装
- SSE 事件类型

### 放到 `agent.py` 的内容

- 新的 AI 主流程
- 多步骤业务编排
- 依赖 `self.llm` / `self.vector_store` / `self.redis_client` 的逻辑

### 放到 `interview_agent.py` 的内容

- 依赖评估结果的模拟面试流程
- 面试题生成与提交评分
- 面试缓存预热与 fallback
- 明显属于“模拟面试智能体”而不是主评估链路的能力

### 放到 `agent_prompts.py` 的内容

- 新 prompt 模板
- prompt 常量
- prompt 构造函数

### 放到 `agent_utils.py` 的内容

- 纯函数
- 数据清洗
- 文本格式化
- 降级函数
- 与 agent 状态弱相关的通用逻辑

### 放到 `models.py` 的内容

- 新 schema
- 新 ORM 模型

## 七、面试时你可以怎么讲这次重构

你可以这样说：

“我在项目迭代中发现 `agent.py` 逐渐变长，于是做了一次职责收敛，但没有过度拆分。我的做法是保留 `ResumeAgent` 作为 AI 业务编排核心，把 prompt 模板抽到 `agent_prompts.py`，把纯工具函数、降级逻辑和结构化清洗抽到 `agent_utils.py`。这样既降低了主文件复杂度，也避免过早拆成很多 service 文件导致维护成本上升。这次调整的目标不是追求形式上的分层，而是让主流程更清晰、模块职责更稳定、后续扩展更容易。”

这段回答会比单纯说“我做了模块拆分”更专业。

---

## 八、最终结论

当前这套分层是合理的：

- 不臃肿
- 不过度设计
- 主流程清晰
- 后续可继续演进

一句话概括：

**现在的后端结构已经达到了“适合当前项目规模的最佳实践平衡点”。**
